# 吞吐优先的两版对比说明

## 1. 文档目的
本文档用于记录当前项目在“时序优化”和“总性能优化”之间的一次实际对比，明确第一版与第二版的差异、收益和问题，并给出后续收敛原则。

本文档中的“总性能”主要指：

- `CoreMark` 跑分对应的真实吞吐
- 在 FPGA 可实现主频下的每秒完成次数

因此，后续优化不能只看 `WNS` 是否改善，而要同时看：

- `CoreMark cycle` 是否恶化
- 可达主频是否提升
- 二者结合后的总吞吐是否真正变好

## 2. 两个版本的定义

### 第一版
第一版指“未进行这轮控制路径时序优化之前”的稳定版本。

该版本的特点是：

- 功能验证稳定
- `CoreMark` 周期数较好
- FPGA 实现频率未达到 `100MHz`
- 但整体吞吐表现优于当前第二版

### 第二版
第二版指“本轮控制路径时序优化之后”的版本。

本轮 RTL 改动主要集中在：

- [cpu_top.v](D:\Codex project\RISC-V CPU\rv32im_low-power\rtl\cpu_top.v)
- [ex_stage.v](D:\Codex project\RISC-V CPU\rv32im_low-power\rtl\ex_stage.v)

主要优化方向包括：

- 拆分 `redirect_valid` 公共控制
- 引入 `frontend_redirect_valid`
- 引入 `id_ex_pc_plus4`
- 引入 `fast_redirect_pc`
- 在 `EX` 级复用 `pc_plus4`
- 显式化 `predict_target_mismatch`
- 尝试降低 `redirect / branch recovery / flush` 控制锥深度

这版优化的结果是：

- FPGA 时序略有改善
- 但 `CoreMark` 周期数明显变差
- 综合后总吞吐不升反降

## 3. 两版关键数据

### 第一版数据

- `CoreMark iterations=10`：`4,218,156 cycles`
- 单次平均：`421,815.6 cycles`
- FPGA 实现结果：`WNS = -1.732ns`
- 等效可达主频约：`85.2MHz`

### 第二版数据

- `CoreMark iterations=10`：`4,515,989 cycles`
- 单次平均：`451,598.9 cycles`
- FPGA 实现结果：`WNS = -1.520ns`
- 等效可达主频约：`86.8MHz`

### 辅助观察

- 第二版相较第一版，`WNS` 改善约 `0.212ns`
- 但 `CoreMark` 单次周期上升约 `29,783.3 cycles`
- `CoreMark` 周期退化约 `7.1%`
- 矩阵乘法 benchmark 基本不变，说明问题主要体现在 `CoreMark` 更敏感的控制流和前端行为上

## 4. 与目标值的差距

当前评估目标为：

- 参考目标主频：`100MHz`
- 参考目标吞吐：`250 次/秒`

吞吐计算方式为：

`吞吐 ≈ 工作频率 / 单次运行周期数`

### 4.1 按统一的 `100MHz` 口径比较

#### 第一版

- 预测吞吐：`100,000,000 / 421,815.6 ≈ 237.07 次/秒`
- 距离 `250 次/秒`：少 `12.93 次/秒`
- 相对差距约：`5.17%`

#### 第二版

- 预测吞吐：`100,000,000 / 451,598.9 ≈ 221.44 次/秒`
- 距离 `250 次/秒`：少 `28.56 次/秒`
- 相对差距约：`11.43%`

### 4.2 结合当前各自可达主频比较

#### 第一版

- 预测吞吐：`85,200,000 / 421,815.6 ≈ 201.98 次/秒`
- 距离 `250 次/秒`：少 `48.02 次/秒`
- 相对差距约：`19.21%`

#### 第二版

- 预测吞吐：`86,800,000 / 451,598.9 ≈ 192.21 次/秒`
- 距离 `250 次/秒`：少 `57.79 次/秒`
- 相对差距约：`23.12%`

## 5. 结果分析

从这次两版对比可以得到一个很明确的结论：

第二版虽然在时序上略有改善，但改善幅度不足以覆盖 `CoreMark` 周期数恶化带来的损失，因此总性能退步。

这说明本项目当前阶段不应再把“单独压 `WNS`”作为唯一目标，而应切换为“吞吐优先”的收敛原则。

也就是说，后续任何 RTL 修改，都必须同时满足以下判断逻辑：

- 如果 `WNS` 变好，但 `CoreMark` 变差更多，则该优化不成立
- 如果 `WNS` 提升有限，而 `CoreMark` 明显退化，则该优化应回退
- 只有当“频率提升收益”大于“周期数损失”时，该优化才值得保留

## 6. 当前结论

截至本文档记录时，可以把两个版本的定位总结为：

- 第一版：时序未闭合，但总性能更好，是更合理的性能基线
- 第二版：时序略有改善，但 `CoreMark` 跑分退化，不适合作为最终性能收敛方向

因此，后续优化策略应调整为：

- 保留确实有助于时序、且不伤害吞吐的改动
- 回避或回退那些“只改善少量时序、却明显拉低 `CoreMark`”的路径
- 后续优先寻找对 `CoreMark` 友好的控制路径、前端路径和前递路径优化方法

## 7. 后续收敛原则

后续版本的优化不再单独追求更小的负 `WNS`，而应统一遵循下面的原则：

1. 以 `CoreMark` 总吞吐作为主要优化目标。
2. 以 `100MHz` 参考口径和“当前可达主频”口径同时评估结果。
3. 允许为了频率和吞吐适当增加少量 `LUT/FF` 冗余。
4. 不接受“时序略有改善但跑分明显下降”的优化。
5. 后续收敛应优先寻找“频率提升且周期基本不变”或“周期下降且频率不恶化”的方向。

## 8. 一句话总结

这次对比说明：对于本项目，`时序改善` 不等于 `总性能提升`；真正应优先收敛的是 `CoreMark 吞吐`，而不是单独的 `WNS` 数字。
